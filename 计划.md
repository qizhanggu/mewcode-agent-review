# LocalDesk Agent 改造计划

> 状态：**Phase 0 执行中：正在完成基线审计、架构设计与最小验证；尚未改动业务逻辑。**
>
> 项目来源说明：现有仓库包含 MewCode Python Coding Agent 学习型底座。本计划只描述后续个人完成的 Desktop Agent 改造；不将原始底座宣称为从零开发成果，也不复制参考项目的代码、界面、品牌或文案。

## 1. 结论与最终定位

将项目定位为：

> **LocalDesk Agent：面向个人授权目录的 Local-first Desktop Agent。**
> 它将自然语言任务转为可审阅的执行计划，检索本地资料、生成 Markdown 交付物、整理文件；所有越权和有副作用操作均受到强制策略约束，并留下可回放任务轨迹。

这里的 “Desktop” 指在用户机器、本地授权目录中执行任务，**不等同于**第一阶段就制作 Electron GUI、控制鼠标键盘或浏览器。v1 继续使用现有 Textual 终端界面和非交互 CLI；这样能把有限时间投入到 Agent Runtime、安全与评估，而不是另起一套前端工程。

## 2. 已完成的输入审计

### 2.1 MewCode 代码底座

已逐文件建立源码索引：153 个 Python 文件，约 23,506 行 Python 源码；测试覆盖 Agent、上下文、会话、权限、MCP、Skill、子 Agent、团队与 Worktree。当前环境未安装 `pytest`，因此尚未获得基线测试的运行结果；开始实现前必须先通过 `uv sync --group dev` 或等价方式安装并记录基线。

当前一次交互的主链路是：

```text
__main__.py / app.py
  -> create_default_registry()
  -> PermissionChecker + PathSandbox + RuleEngine
  -> Agent.run(conversation)
  -> LLM stream
  -> ToolRegistry / Tool.execute()
  -> ToolResultEvent / PermissionRequest
  -> Textual UI、Session JSONL、文件历史
```

可复用资产：

| 模块 | 结论 | 原因 |
|---|---|---|
| `agent.py` | 保留并适配 | 已有 LLM 流式循环、工具批处理、用户确认事件、错误回流、上下文压缩。 |
| `tools/base.py`、`tools/__init__.py` | 保留并扩展 | `Tool`、Pydantic 参数模型和 Registry 是 Desktop Skill 的合适接口。 |
| `client.py`、`conversation.py`、`serialization.py` | 保留 | 提供多模型协议适配与对话消息序列化。 |
| `context/`、`memory/session.py` | 保留 | 分别处理长上下文和 JSONL 会话恢复；不要把它们误当作知识库。 |
| `permissions/` 的规则与危险命令检测 | 选择性复用 | 可借用规则层和路径解析思路，但不能直接作为新安全边界。 |
| `app.py`、`permission_dialog.py` | 保留，后续轻改 | Textual 已能呈现工具调用、确认与中断，v1 无需 Electron。 |
| `mcp/`、`skills/`、`hooks/` | 暂不改动 | 作为后续扩展点，不进入第一个闭环。 |

冻结但不删除的 Coding Agent 专属模块：

| 目录 / 模块 | 处理方式 |
|---|---|
| `worktree/`、`tools/enter_worktree.py`、`tools/exit_worktree.py` | Desktop 模式不装配、不改写；保留原 Coding 模式。 |
| `tools/edit_file.py`、原 `WriteFile`、`Grep`、`Glob` | Coding 模式继续使用；Desktop 模式使用新的受策略约束工具。 |
| `teams/`、`agents/`、`tools/agent_tool.py`、团队工具 | v1 禁用，后续不以 Multi-Agent 作为卖点。 |
| `tools/bash.py`、OS sandbox | v1 Desktop 模式不注册；后续仅考虑白名单命令模板，绝不开放任意自然语言 Shell。 |

必须正视的现有边界：

1. 现有 `PathSandbox` 将越出根目录的路径返回为 `ask`，用户可继续批准；Desktop Agent 必须把越权路径变成不可批准的 `deny`。
2. 权限规则的 `extract_content()` 目前只提取单个字段，如 `file_path`；文件移动至少同时包含源与目标，必须有新的多路径校验。
3. `Bash` 使用 `create_subprocess_shell()` 执行任意字符串；“工作目录在授权目录内”不能约束绝对路径、网络、重定向或子进程。
4. 现有 `TraceManager` 面向子 Agent 图，不是持久化的任务审计记录；Desktop 任务应新建独立 Trace Store。
5. 现有 `FileHistory` 是“编辑前文件备份”，适合借鉴，不足以表示移动、重命名、创建产物和确认决定。

## 3. v1 的能力边界

### 要做

1. 用户配置一个或多个授权资料目录，以及一个授权输出目录。
2. 读取 Markdown、TXT、文本型 PDF，检索时返回 `文件路径 + 页码/片段编号 + 摘要`。
3. 基于检索内容生成 Markdown 周报、项目报告、岗位/简历建议，并先写入应用 staging 区。
4. 扫描文件、生成分类方案；用户看见精确变更清单后，才能移动或重命名。
5. 每个任务保留状态、计划、动作、策略判断、用户确认、结果、失败和产物路径。

### 不做

- 鼠标键盘控制、浏览器自动化、远程服务、日历邮件、云盘整合。
- Multi-Agent 团队、Workflow 画布、定时任务。
- 任意 Shell、删除文件、覆盖已有文件、扫描型 PDF OCR、复杂 DOCX 解析。
- 新建 Electron/React 客户端。

DOCX 导出是 v1 完成且评估通过后的可选增量；它不是首版验收条件。

## 4. 目标架构

```text
CLI / Textual UI
  -> DesktopTaskService
      -> TaskPlanner（输出结构化计划和 Action 草案）
      -> DesktopPolicyGuard（独立于 LLM；检查所有路径与风险）
      -> DesktopToolRegistry
          -> Knowledge Skill（扫描、抽取、检索、来源）
          -> Document Skill（生成内容、staging、提交产物）
          -> File Skill（扫描、dry-run、移动/重命名）
      -> TaskTraceStore（JSON 事件流 + Artifact 清单）
```

设计规则：

- Planner 只提出 `PlannedAction`，不直接触及文件系统。
- Policy Guard 审查 Action 的**每一个**输入和输出路径；越权直接拒绝，不展示“继续执行”按钮。
- 写入先到 `.mewcode/desktop/staging/<task_id>/`；确认时展示内容摘要、目标路径、哈希和精确文件变更，批准后才提交到 output 目录。
- 文件整理先产生 `FileOperation` 列表并 dry-run；确认后串行执行，不允许覆盖同名目标；执行日志能支持“仅对成功 move/rename 生成回滚清单”。
- 一次任务只有一个 `task_id`，状态为 `draft -> planned -> awaiting_confirmation -> executing -> succeeded | failed | cancelled`。
- 工具失败是可记录的事件与可见结果，不能被异常吞掉；任务取消后不能执行尚未开始的 Action。

## 5. 建议目录和数据模型

在 `mewcode/desktop/` 新建独立边界，不将 Desktop 逻辑塞进 1,300 行的 `agent.py`：

```text
mewcode/desktop/
  __init__.py
  models.py          # Task、TaskStatus、PlannedAction、Artifact、TraceEvent
  service.py         # 创建/恢复/推进任务，衔接现有 Agent 事件
  planner.py         # Planner prompt、计划校验、Action 结构化解析
  policy.py          # DesktopPolicyGuard、所有路径/风险校验
  trace_store.py     # task.json 与 events.jsonl，原子写入
  workspace.py       # 授权根目录、输出根目录、规范化 Windows 路径
  skills/
    knowledge.py     # 文本抽取、分块、词法检索、引用
    document.py      # Markdown 草稿、staging、提交产物
    files.py         # 清单、分类计划、执行与回滚清单
```

关键模型建议使用 dataclass 或 Pydantic，但必须显式序列化：

```text
Task(task_id, user_query, status, plan, actions, artifacts, error, created_at)
PlannedAction(action_id, skill, tool, args, risk, status, preview)
Artifact(kind, staged_path, final_path, sha256, summary)
TraceEvent(sequence, timestamp, event_type, payload)
```

## 6. 分阶段实施计划

### Phase 0：建立可验证基线与改造保护线（执行中）

目的：确保后续工作可回归，而非在原底座上“边改边猜”。

1. 安装开发依赖，运行 `python -m pytest -q`，记录通过数、失败数和 Python/依赖版本。
2. 新建 `docs/current_architecture.md`：写入本计划第 2.1 节的主链路、保留/冻结模块和关键限制。
3. 为现有 Coding 模式补一个最小启动/Registry 快照测试，保证 Desktop 改造不会默认装配 Worktree、Teams、Bash。
4. 不删除任何 MewCode 原模块；新增 `--desktop` 模式或同等独立入口。

当前验证记录见 [`docs/baseline_validation.md`](docs/baseline_validation.md)：在仓库内受控临时目录运行时，原始基线为 **542 passed、8 failed、1 skipped**；8 项均已分类为原始快照/Windows 兼容问题，不能归因于 LocalDesk 改造。

验收：原测试可重复运行；新入口尚未注册危险工具；Git 有独立 `docs(audit): ...` 提交。

### Phase 1：任务状态机、授权工作区和 Trace

目的：先让任务“可控、可恢复、可解释”，再加能力。

1. 实现 `desktop/models.py`、`workspace.py`、`trace_store.py`。
2. 新增 Desktop 配置段：授权资料根目录列表、唯一 output 根目录、任务数据目录；仅用配置或 CLI 参数设置，首版不做文件夹选择 GUI。
3. 实现 `DesktopPolicyGuard`：
   - `read`：仅允许授权资料根目录和任务 staging；
   - `write/move/rename`：仅允许 staging 或 output 根目录，且需要确认；
   - 越权、符号链接逃逸、覆盖目标：直接拒绝；
   - `shell/delete/network`：Desktop v1 一律拒绝。
4. 实现任务创建、状态转移与 JSONL Trace；在 Textual 中最少展示 task id、状态和确认请求。

验收：不调用 LLM 的单元测试可覆盖状态转移、Windows 路径归一化、`..` 路径、软链接逃逸、双根目录移动、拒绝覆盖与 Trace 恢复。

### Phase 2：安全的“资料 -> Markdown 交付物”闭环

目的：形成第一个可演示、可评估的 Agent 任务。

1. 实现 Knowledge Skill：Markdown/TXT/PDF 文本抽取、稳定 chunk id、轻量词法检索；PDF 只支持可提取文本，结果必须带文件、页码或片段定位。
2. 实现 Document Skill：根据任务要求和引用上下文创建 Markdown 草稿，写入 staging，而非直接写入用户目录。
3. 引入 Planner：输出可验证的计划（任务目标、要检索的根目录、拟生成产物、风险动作），不再引入第二个独立“Skill Router”LLM。
4. 用户确认后，Document Skill 将 staging 文档提交至 output；Trace 保存引用、预览、哈希和最终路径。

首个验收任务：

> “读取授权目录中的 Markdown/PDF 会议纪要和任务记录，生成带来源引用的本周周报到 output 目录。”

验收：报告存在、所有引用均能定位、未确认时 output 目录无新文件、确认记录与产物哈希可在 Trace 中回放。

### Phase 3：文件整理闭环与可逆操作

目的：体现真正的桌面执行能力，而不牺牲可控性。

1. 实现 File Skill 的只读扫描：文件清单、类型、大小、修改时间。
2. 实现分类计划生成：只输出 `source -> destination`，不执行。
3. Policy Guard 对每一个 source/destination 检查；展示总数、冲突和精确变更清单。
4. 确认后串行 move/rename，失败停止或按明确策略继续；写入 operation journal。
5. 为成功 move/rename 提供 `rollback` 计划和受同一策略约束的回滚执行；不支持删除与覆盖。

验收任务：

> “将授权 Downloads 样例目录的 PDF、图片、压缩包按类型整理；先展示计划，确认后执行。”

验收：未确认零副作用；确认后路径全部符合计划；冲突不覆盖；回滚后文件回到原位；越权目标 100% 被拒绝。

### Phase 4：评估、展示与项目叙事

目的：把能力变为可证明的求职材料。

1. 建立 20–30 条无敏感数据的任务评估集和固定样例资料。
2. 指标：
   - Task Success Rate；
   - Citation Validity；
   - File Plan Validity；
   - Confirmation Compliance（未确认即产生副作用应为 0）；
   - Unsafe Action Block Rate（越权/删除/Shell 用例应为 100%）；
   - 平均工具数、时延、失败恢复率。
3. 建立 deterministic fake LLM 集成测试，避免日常测试依赖真实 API；真实模型仅用于最终评估和 Demo。
4. 更新 README：来源说明、独立改造范围、架构、威胁模型、评估结果、安装/运行方式、非目标能力。
5. 录制三个短 Demo：带引用周报、文件分类确认、越权/危险动作拒绝。

验收：每次评估可复现；README 不夸大；Demo 使用公开构造样例；每一阶段一个可读 Git 提交。

### Phase 5：只在 v1 验收后讨论的增量

- DOCX 输出（只新增生成，不先做解析）。
- 受限 Shell：固定命令模板 + 参数 schema + 无网络 + 再次确认；不接受任意命令字符串。
- MCP：只为已确认的本地工具提供扩展，不把 MCP 当作项目核心亮点。
- 更丰富的 Textual Task Timeline；Electron 仅在已有稳定 Runtime 后另立子项目决策。

## 8. 首次实施前的决策门

在开始 Phase 1 前，先确认这三个约束不变：

1. v1 的正式名称和 README 定位采用 “Local-first Desktop Agent”，而不是 “Claude/Coworker 复刻品”。
2. v1 不注册 `Bash`、Worktree、Teams、任意删除工具；这些模块仅保留在旧 Coding 模式。
3. 第一个端到端验收场景固定为“本地资料检索并生成带引用 Markdown 周报”，文件整理作为第二闭环。

如果三项确认，则按 Phase 0 -> Phase 1 顺序推进；不跳过基线测试、任务状态机和策略测试，直接实现 RAG 或 UI。
