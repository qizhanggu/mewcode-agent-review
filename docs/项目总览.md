# LocalDesk Agent 项目总览

## 结论：Desktop v1 算完成了吗？

**算完成了核心 v1。** 更准确地说，它是一个“受控本地任务 Agent 的可验证 MVP”，不是一个全能桌面助手。

已经完成：本地资料检索、带引用 Markdown 草稿、staging 与哈希、二次确认交付、任务 Trace、Grounded LLM 结构化渲染约束，以及文件整理/回滚核心工作流。未完成或明确不做：任意 Shell、浏览器、鼠标键盘自动化、MCP、多 Agent、Electron、云盘邮件日历、真实 GUI 控制。

仍待补的一项是：本机未配置模型 provider，所以真实 LLM 的 5–10 条指标还没跑；不能把 fake LLM 测试当成真实模型效果。

## 大白话：这个项目现在能干什么

把它想成一个很谨慎的本地助手。

你把一个资料文件夹授权给它，说“帮我汇总会议纪要”。它先只能看这个文件夹，找到相关内容并标好来源；然后在自己的草稿区写报告，不会直接往你的输出目录塞文件。你检查草稿后再确认，它才交付最终 Markdown。

如果用 LLM，它也不是“想干什么就干什么”：它只能看到已经检索到的几段文字，每段有编号；写报告时每一段都必须说自己依据了哪个编号。瞎编编号、不给依据、超时或格式不对，系统就直接拒绝，不生成文件。

文件整理模块像“先给你一张搬家清单”：它会先列出每个文件从哪里搬到哪里；你确认后才按顺序移动。移动到一半出错，它会停下来，记住前面成功搬了哪些；回滚时也只搬回这些成功过的文件，并且要你再确认一次。

## 可直接写入简历的项目介绍

**LocalDesk Agent｜受控本地资料与文件任务 Agent｜Python、LLM Tool Calling、Pydantic、Textual、JSONL Trace**
基于学习型 Python Coding Agent 底座完成独立 Desktop Agent 改造，面向用户授权目录提供本地资料检索、带引用 Markdown 交付和受控文件整理能力；围绕“模型输出不可信、文件副作用高风险、执行过程不可追溯”设计任务状态机、deny-first 权限策略、staging + SHA-256 二次确认和 JSONL Trace 闭环。

- 实现 MD/TXT/文本型 PDF 轻量检索，结果定位到文件行号或 PDF 页码；构建 Grounded LLM 渲染约束，模型仅可消费带 citation_id 的检索片段并输出结构化 JSON，虚构引用、无引用结论、格式异常和超时均拒绝暂存。
- 实现“草稿暂存 → 用户确认 → output 交付”链路，确认前 output 零副作用、确认时校验草稿哈希；覆盖越权路径、覆盖写入、Shell、删除、篡改草稿等安全场景。
- 实现并测试受控文件整理与回滚核心工作流：dry-run 展示精确变更，串行执行记录 operation journal；中途失败停止后续操作，回滚仅针对已成功项且需独立确认；确定性测试 **34 passed，1 skipped**。

> 面试口径：文件整理是“已实现并测试的核心工作流模块”，不是已完成的 CLI 用户入口；真实 LLM 指标待配置 provider 后补跑。
