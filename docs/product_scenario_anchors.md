# LocalDesk 产品场景锚点

> 本文描述的是 LocalDesk 的目标产品场景，不代表当前仓库已经完成全部能力。
>
> 一句话定位：**跟电脑说一句，把一件办公室杂活办完。**

## 场景一：把《猫猫快递》今天的玩家差评处理掉

### 一句话任务

> 老板说：“把今天的玩家差评处理掉，严重 Bug 登记进系统，提交前让我确认。”

这句话就是场景的记忆锚点。其余步骤都是 LocalDesk 在后台完成的工作。

### 场景背景

一家小型独立游戏工作室发布了游戏《猫猫快递》。每天早晨，运营或测试人员都要查看 Steam 评论、客服工单和崩溃日志，整理问题、生成日报，并把严重问题登记进 Bug 系统。

授权目录中包含：

```text
今日客服工单.xlsx
游戏崩溃日志.txt
当前版本说明.docx
上周玩家反馈报告.docx
玩家截图/
```

公开信息来源包括游戏的 Steam 评论页面和社区讨论区。

### LocalDesk 要完成什么

1. 收集最近 24 小时的 Steam 评价和社区反馈。
2. 读取本地客服工单、崩溃日志和版本说明。
3. 合并重复问题，区分 Bug、玩法建议、价格抱怨和无关情绪。
4. 根据影响人数、问题严重程度和证据划分 P0/P1/P2。
5. 生成带来源的 `玩家反馈日报.docx`。
6. 在白名单 Bug Tracker 中填写严重问题。
7. 真正提交 Bug 前请求用户确认，提交后验证 Bug 编号。
8. 通知用户任务已经完成，并给出日报和 Bug 链接。

### 为什么不是一段固定脚本

- 每天的评论数量、语言和表达方式不同。
- 同一个问题可能同时出现在评论、工单和日志中，需要语义合并。
- 玩家说“进不去”可能是客户端 Bug，也可能是服务器维护。
- 页面打不开、资料缺失或来源冲突时，需要改变执行路径或询问用户。
- Skill 负责反馈分类、日报格式和 Bug 填写规则；Agent 负责选择和组合 Skill、处理异常并决定何时请求人工确认。

### 可验证的最小 Demo

- 准备一套包含已知答案的评论、工单、日志和版本说明。
- 预设至少三类问题，并包含重复反馈、无关抱怨和一条证据冲突。
- 系统正确生成日报，并只为满足条件的问题创建 Bug 草稿。
- 用户拒绝时不提交；用户确认后提交并验证结果。
- 第二次运行时能够沿用用户确认过的报告偏好。

## 场景二：暴雨明天到，把十二家奶茶店快过期的鲜奶救下来

### 一句话任务

> 区域经理说：“明天台风，三家店要停业，把快过期的鲜奶和珍珠调到还能营业的门店，别走封闭道路，调拨单提交前给我看。”

这句话就是第二个场景的记忆锚点。

### 场景背景

一家奶茶连锁品牌在同一城市有 12 家门店。晚上 17:40，气象部门发布暴雨预警，其中三家门店第二天暂停营业。部分鲜奶将在 24 小时内到期，如果不及时调拨就只能报废。

授权目录中包含：

```text
十二家门店库存.xlsx
近七日销量.csv
原料保质期规则.docx
配送车辆与容量.xlsx
上次调拨记录.xlsx
```

公开信息来源包括气象预警和道路封闭公告。

### LocalDesk 要完成什么

1. 读取 12 家门店的库存、保质期、销量和车辆容量。
2. 浏览天气预警与道路封闭公告。
3. 找出即将停业且存在临期原料的门店。
4. 估算其他门店第二天能够消耗的数量，避免把原料调到同样卖不完的门店。
5. 排除道路封闭、车辆超载和超过保质期的方案。
6. 生成 `暴雨应急调拨方案.xlsx` 和 `区域经理简报.docx`。
7. 在白名单调拨系统中生成调拨单草稿。
8. 显示“从哪家店、调多少、送到哪家店、为什么”，用户确认后再提交。

### 一个具体测试例子

- 猫爪路店明天停业，有 22 升鲜奶将在 20 小时后到期。
- 月亮湾店明天营业，预计缺少 14 升鲜奶，能够接收其中 14 升。
- 火车站店也能接收，但通往该店的道路已经封闭。
- 剩余 8 升没有安全可行的去向，系统应明确列为预计损耗，不能为了“全部救下”编造不可行方案。

正确结果至少应满足：不走封闭道路、不超过接收门店需求、不超过车辆容量、不使用已经过期的原料，并且提交调拨单前必须确认。

### 为什么需要 Agent

这个任务有稳定的业务目标，但每天的库存、天气、停业门店、道路和车辆都不同。确定性的计算和约束检查应当由普通代码完成；Agent 负责理解临时指令、选择资料、处理缺失或冲突信息、组合计算与文档 Skill，并在无法安全决定时请求人工处理。

## 两个场景如何用于项目开发

### 主场景

先把“猫猫快递玩家反馈处理”作为 Hero Demo 做穿。它与当前已经完成的本地检索、网页读取、DOCX、Trace 和白名单 UIA 能力更接近。

### 迁移场景

完成主场景后，再实现“暴雨前奶茶原料调拨”的受控版本。它不是为了再造一个独立项目，而是验证同一个 Runtime 能否更换领域 Skill 后处理另一类任务。

如果第二个场景必须重写整个 Runtime，说明第一版对主场景硬编码过多；如果只需增加库存计算、约束检查和调拨文档等领域 Skill，说明平台抽象基本成立。

## 推进原则

不要先堆出一个功能很多但没有完整业务闭环的“丐版 WorkBuddy”，也不要等所有产品问题都想清楚才写代码。采用以下顺序：

1. 冻结已经完成的通用底座，停止增加与场景无关的框架。
2. 用固定评测数据完成猫猫快递主场景的手动触发闭环。
3. 加入资料缺失、来源冲突、页面失败和用户拒绝等动态分支。
4. 加入用户反馈后的版本修改、显式偏好记忆和再次运行。
5. 将成功任务保存为可重复流程，再增加定时执行和完成通知。
6. 用奶茶调拨场景进行迁移验证，区分通用 Runtime 与领域 Skill。

项目的验收重点不是“工具数量”，而是：一句自然语言任务能否在变化输入下完成真实产物与受控操作，失败时能否停下、解释和恢复。

## 给执行 AI 的阶段指令

> 现有 Runtime、Policy Guard、Trace、DOCX、网页读取、文件操作和 UIA 基线暂时冻结，不要继续无目标扩建通用能力，也暂时不要围绕求职材料场景继续深化。接下来以本文中的“猫猫快递玩家反馈处理”作为唯一 Hero Demo：用户只下达“把今天的玩家差评处理掉，严重 Bug 登记进系统，提交前让我确认”，系统需要串联公开评论或可复现网页、本地工单与日志、反馈归并与分级、带来源 DOCX 日报、白名单 Bug Tracker 草稿、确认提交、结果验证和完整 Trace。先使用固定、可标注答案的合成数据完成可重复闭环，再加入页面不可用、资料缺失、来源冲突、重复反馈和用户拒绝等失败分支。确定性分类规则、约束检查、文件写入和提交验证尽量使用普通代码或独立 Skill；Agent 只负责意图理解、动态规划、Skill 选择、非结构化判断、异常处理和人工确认。第一阶段完成后，请汇报真实能力、完整 Demo Trace、固定数据集上的任务级结果、失败案例，以及哪些模块属于通用 Runtime、哪些属于猫猫快递领域 Skill。不要先实现奶茶调拨场景；它留作主场景完成后的迁移验证。
